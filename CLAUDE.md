## Project Overview

BengalClub is a Laravel 12 membership club management application with separate admin and member portals. The admin panel uses AdminLTE 3, while the frontend/member portal uses Vite with Tailwind CSS 4. The application uses MySQL as the primary database.

## Development Commands

### Initial Setup
```bash
composer setup
```
This runs the full setup: installs dependencies, copies `.env.example` to `.env`, generates app key, runs migrations, and builds frontend assets.

### Development Server
```bash
composer dev
```
Starts three concurrent processes:
- Laravel development server (`php artisan serve`)
- Queue worker (`php artisan queue:listen --tries=1`)
- Vite dev server with HMR (`npm run dev`)

### Testing
```bash
composer test                      # Run full test suite
php artisan test --filter=TestName # Run specific test
```

PHPUnit uses SQLite in-memory database (see [phpunit.xml](phpunit.xml)).

### Code Quality
```bash
./vendor/bin/pint                  # Fix code style issues
./vendor/bin/pint --test           # Check without fixing
```

### Database
```bash
php artisan migrate                # Run migrations
php artisan migrate:fresh --seed   # Reset and seed (AdminSeeder creates admin accounts)
```

### Building Assets
```bash
npm run build   # Production build
npm run dev     # Development with HMR
```

## Architecture

### Authentication (Two Guards)

Two separate auth guards — this is fundamental to the routing logic:
- `web` guard — frontend `User` model (members)
- `admin` guard — `Admin` model, used as `auth:admin` on all admin routes

All admin routes are under `/admin` prefix with `auth:admin` middleware. Member-authenticated routes are under `/member` prefix.

All routes live in a single [routes/web.php](routes/web.php) — there is no `api.php`. Two helper routes (`/storage-link`, `/run-migrations`) run artisan commands over HTTP for deployment; treat them as deploy-only.

### Middleware (custom)

- `member.not-suspended` → `CheckMemberSuspension` — applied to all authenticated member routes. Logs out and redirects suspended members, but **skips the check while an admin is impersonating** (session key `impersonating_admin_id`).
- Guest/authenticated redirects are path-aware (configured in [bootstrap/app.php](bootstrap/app.php)): `admin/*` requests redirect to `admin.login`/`admin.dashboard`, everything else to the member equivalents.

### Controllers Structure

Controllers are split into three namespaces (admin and frontend often have same-named controllers, imported with aliases in `web.php`):
- `App\Http\Controllers\Admin\` — ~30 admin controllers (CRUD plus report/approval flows)
- `App\Http\Controllers\Frontend\` — member auth, forgot password, dashboard, profile, shop, donations, Tan Samiti, event registration
- `App\Http\Controllers\` — public-facing: `FrontEndController`, `ContactController`, `MembershipApplicationController`

### Key Business Flows

**Membership Application Flow:**
1. Public user submits application form with photo, NID, and reference info (`MembershipApplicationController`)
2. Admin reviews and approves/rejects via `Admin\MembershipApplicationController`
3. On approval: `User` and `UserProfile` records are created, installments generated automatically
4. Member receives welcome email; can log in to member portal

**Membership Installments & Payment Requests:**
- Membership fees are split into installments (`MembershipInstallment` model), each tracking amount, due date, status (pending/completed/overdue).
- Primary flow is member-driven: a member submits a payment (proof, txn id, chosen `PaymentMethod`) via `Frontend\MemberInstallmentController` — this sets `member_submitted_at` (status stays `pending` = "submitted/awaiting approval"). Admin then approves/rejects in `Admin\PaymentRequestController`, which flips status to `completed` and writes a `Transaction` ledger row.
- An admin can also mark installments complete directly via `Admin\MembershipInstallmentController` (writes a `MembershipTransaction`).
- **Two ledger models exist:** `Transaction` (general financial ledger — payment requests, application fees, tied to `PaymentMethod` and `recorded_by_admin_id`) vs `MembershipTransaction` (older per-installment record). New work should prefer `Transaction`.
- Membership categories with `optional_installment` treat installments as voluntary donations rather than dues.

**Tan Samiti (rotating savings / committee):**
- A `TanSamiti` plan has a `monthly_amount`, `total_cycles`, and members (`TanSamitiMember`). Each member gets one `TanSamitiInstallment` per cycle, auto-generated by `TanSamiti::generateInstallmentsForUser()` (regeneratable after plan edits).
- Members join and submit per-cycle payments (member portal); admins approve/reject in `Admin\TanSamitiController`.
- If `enable_lottery_draw` is set, `TanSamitiDrawController` runs a spin to pick one winner per cycle from members who haven't won yet (`eligibleForDraw()`), recorded as a `TanSamitiDraw`.

**Shop / Orders:**
- Public can browse `/shop` and place an `Order` (with `OrderItem`s) for a `Product` — no login required. Admins manage products and update order status in `Admin\ProductController` / `Admin\OrderController`. Members see their own orders at `/member/orders`.

**Donations & Expenses:**
- Public donation form (`/donate`) creates a `Donation` against a `DonationCategory`; admins review/update status. `Expense` records outflows. `Admin\DonationReportController` aggregates donations vs expenses for financial reporting.

**Event Registration:**
- Public registers for an event (`Frontend\EventRegistrationController`); admins approve/cancel in `Admin\EventRegistrationController`. Approval/cancellation send `EventRegistrationApprovedMail` / `EventRegistrationCancelledMail`.

**Invitation System:**
- Admins send invitations with a unique `invite_id`
- Invitations can specify membership category and a custom application fee (overrides global `application_fee` from `SiteSetting`)

**Admin Impersonation & Suspension:**
- Admins can impersonate members (stored in session) via `POST /admin/registered-members/{id}/impersonate`; stop via `POST /member/stop-impersonating`.
- Admins can suspend/unsuspend and extend membership from `Admin\RegisteredMemberController`. Suspended members are blocked by `CheckMemberSuspension` — but impersonation bypasses this so admins can still inspect a suspended account.

**Password Reset:**
- OTP-based (not Laravel's default token-based) — stored in `password_reset_otps` table
- Flow: send OTP → verify OTP → reset password

**Event Notifications:**
- When an event is created, `SendEventNotificationEmail` job is dispatched (queued) to email all members
- Queue must be running (`php artisan queue:listen`) for emails to send

**Inquiry Rate Limiting:**
- Contact form (`Inquiry` model): max 2 submissions per 24 hours with minimum 12-hour gap between them (IP-based, enforced in `canSendInquiry()`)

### Models — Important Patterns

- `SiteSetting` and `SeoSetting` are singletons — always `firstOrCreate(['id' => 1])`, ID always 1
- `ViewServiceProvider` shares both with all views automatically — access as `$siteSetting` and `$seoSetting` in any Blade template
- JSON columns: `Event::gallery_images`, `Facility::features`, `MembershipCategory::features`, `BoardDirector::social_links` — stored as JSON arrays
- `MembershipApplication` has scopes: `pending()`, `accepted()`, `rejected()`
- `AboutUs` and `VisionMission` are also edited as singletons (single edit/update screen, no index/CRUD)

### Mail & Queue

- Mailables live in `app/Mail/` (application received/approved, member welcome, invitation, event created/registration, OTP). Most are sent inline; the bulk **event-creation broadcast** (`SendEventNotificationEmail`) is the one queued job — it needs the queue worker running to deliver.

### Key Packages

- `barryvdh/laravel-dompdf` — PDF generation: membership applications, certificates, Tan Samiti agreements
- `simplesoftwareio/simple-qrcode` — QR codes for member profiles (`/member/{userId}/qr-code`); members also have a vCard download (`/member/{userId}/vcard`)
- `maatwebsite/excel` — member export (`App\Exports\MembersExport`, `GET /admin/registered-members/export`)

### Frontend vs Admin Assets

**Admin Panel** — no Vite, all pre-compiled:
- AdminLTE 3 CSS/JS from `public/dist/`
- Plugins (jQuery, Bootstrap 4, Summernote, SweetAlert2, etc.) from `public/plugins/`
- Dark mode toggle persisted via `localStorage`

**Member/Frontend Portal** — Vite compiled:
- Entry: `resources/css/app.css` + `resources/js/app.js`
- Tailwind CSS 4 via `@tailwindcss/vite` plugin
- Vite ignores `storage/framework/views/**`

### Database

- **Development:** MySQL, database named `bengalclub`
- **Testing:** SQLite in-memory (`:memory:`)
- Session, queue, and cache: all database-backed

### Seeders

- `DatabaseSeeder` — creates a test user (`test@example.com`)
- `AdminSeeder` — seeds admin accounts (run via `migrate:fresh --seed`)
